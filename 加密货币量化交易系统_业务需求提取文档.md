## 第1部分：业务目标（SMART原则）

### 业务目标1：构建面向个人投资者的一站式加密货币量化交易平台

- **Specific（具体）**: 
  - 建设集行情、交易、策略、社区于一体的加密货币量化交易平台
  - 支持主流加密货币交易所接入（Binance、OKX、Hyperliquid等）
  - 提供手动交易盘面和自动化策略交易两种模式
  - 对标竞品：知行量化（zxlh.pro）、米筐、聚宽

- **Measurable（可量化）**: 
  - P0阶段用户注册量目标：1000+
  - 支持交易所数量：≥3家主流交易所
  - 系统可用性：≥99.5%
  - 行情延迟：<500ms
  - 订单执行延迟：<1秒
  - 用户满意度：≥80%

- **Achievable（可实现）**: 
  - 采用成熟的交易所API接入方案
  - 复用TradingView等成熟行情组件
  - 基于Web技术栈快速迭代
  - 渐进式功能开发，MVP先行

- **Relevant（相关性）**: 
  - 加密货币市场持续增长，2025年全球加密货币总市值突破7.5万亿美元
  - 个人投资者对量化交易工具需求旺盛
  - 现有竞品多为机构级产品，个人用户门槛高
  - 解决个人投资者策略验证难、自动化交易门槛高的痛点

- **Time-bound（时限）**: 
  - P0阶段（MVP）：2个月内上线核心功能
  - P1阶段：4个月内完善策略生态
  - P2阶段：6个月内建设社区与增值服务

- **来源**: （引用自用户需求描述"做一个量化交易系统，模仿知行量化"）

## 第2部分：顶层业务用例

### 业务用例1：查看实时行情与技术分析

- **主要参与者**: 投资者（注册用户/游客）
- **利益相关者**: 
  - 投资者：希望获得实时、准确的行情数据，辅助交易决策
  - 平台运营：希望通过优质行情服务吸引和留存用户
  - 交易所：希望增加交易量
- **业务目标**: 
  - 提供专业级行情分析工具
  - 降低用户技术分析门槛
  - 为交易决策提供数据支撑
- **前置条件**: 
  - 系统已接入交易所行情数据
  - 用户已访问行情页面
- **后置条件**: 
  - 成功：用户获得所需行情信息，可进行技术分析
  - 失败：行情加载失败，提示用户稍后重试
- **主要业务流程**:
  1. 用户进入行情页面
  2. 用户选择交易对（如BTC/USDT）
  3. 系统加载实时K线数据
  4. 用户切换K线周期（1分钟/5分钟/1小时/1天等）
  5. 用户添加技术指标（MA/MACD/RSI等）
  6. 用户使用画线工具进行标注
  7. 系统实时更新最新行情
- **扩展流程**:
  - **E1.行情断连**: 系统自动重连，显示断连提示
  - **E2.多图表对比**: 用户可同时打开多个交易对图表
  - **E3.指标参数调整**: 用户可自定义指标参数
  - **E4.图表截图分享**: 用户可保存或分享图表截图
- **业务规则**:
  - BR1: 行情数据延迟<500ms
  - BR2: K线周期支持：1m/5m/15m/30m/1h/4h/1d/1w/1M
  - BR3: 默认显示最近200根K线
  - BR4: 技术指标参数可配置
- **量化指标**:
  - 行情加载时间：<2秒
  - 实时更新频率：≤1秒
  - 图表渲染帧率：≥30fps
- **优先级**: Must Have
- **来源**: （引用自用户需求描述"行情可以类似tradingview，可以看k线，看指标，手动划线等"）

### 业务用例2：手动下单交易（现货+合约）

- **主要参与者**: 投资者（已配置交易所的用户）
- **利益相关者**: 
  - 投资者：希望便捷地执行交易操作，包括现货和合约交易
  - 平台运营：希望用户在平台完成交易
  - 风控团队：希望交易符合风险控制要求，尤其是合约交易的杠杆风控
- **业务目标**: 
  - 提供便捷的交易入口
  - 支持在行情页面直接下单
  - **支持现货交易和杠杆/合约交易两种模式**
  - 保障交易执行的可靠性
- **前置条件**: 
  - 用户已登录
  - 用户已配置交易所API Key
  - 用户账户有足够余额/保证金
  - （合约交易）用户已开通合约交易权限
- **后置条件**: 
  - 成功：订单提交到交易所，用户收到成交反馈
  - 失败：订单提交失败，显示失败原因
- **主要业务流程**:
  1. 用户在行情页面点击"交易"或"下单"
  2. 系统显示下单面板
  3. **用户选择交易模式（现货/合约）**
  4. **（合约模式）用户选择合约类型（USDT永续/币本位永续）**
  5. **（合约模式）用户选择杠杆倍数（1x-125x）**
  6. **（合约模式）用户选择保证金模式（逐仓/全仓）**
  7. 用户选择交易方向（买入/卖出；**合约：做多/做空**）
  8. 用户选择订单类型（市价单/限价单/止盈止损单）
  9. 用户输入数量和价格（限价单）
  10. 系统显示预估成交金额、手续费、**预估强平价格（合约）**
  11. 用户确认下单
  12. 系统调用交易所API提交订单
  13. 系统显示订单状态和成交结果
- **扩展流程**:
  - **E1.余额/保证金不足**: 系统提示不足，禁止下单
  - **E2.订单超时**: 系统提示超时，用户可重试
  - **E3.部分成交**: 系统显示部分成交状态，用户可选择撤销剩余
  - **E4.快捷下单**: 支持一键买入/卖出预设比例
  - **E5.杠杆超限**: 合约杠杆超过平台限制时提示
  - **E6.保证金预警**: 保证金率接近警戒线时提示
  - **E7.强平风险**: 预估强平价格接近当前价格时警告
- **业务规则**:
  - BR1: 下单前必须验证API Key有效性
  - BR2: 市价单立即执行，限价单挂单等待
  - BR3: 订单数量和金额有最小限制（交易所规定）
  - BR4: 高风险操作需二次确认
  - **BR5: 合约交易杠杆倍数受平台最大杠杆限制（默认最高20x，可配置）**
  - **BR6: 合约交易必须设置止损**
  - **BR7: 保证金率低于维持保证金率时触发强平警告**
  - **BR8: 新用户首次合约交易需阅读风险提示并确认**
- **量化指标**:
  - 下单响应时间：<1秒
  - 订单提交成功率：≥99%
  - 成交反馈延迟：<3秒
  - **合约保证金计算延迟：<500ms**
  - **强平价格更新频率：实时**
- **优先级**: **Must Have（现货P0，合约基础P0，合约高级P1）**
- **来源**: （引用自用户需求描述"在行情页面可以手动下单"，扩展自"盘面交易里面，帮我加杠杆或合约交易的功能"）

### 业务用例3：浏览与订阅策略

- **主要参与者**: 投资者（注册用户）
- **利益相关者**: 
  - 投资者：希望找到适合自己的交易策略
  - 策略提供方：希望策略被更多用户使用
  - 平台运营：希望通过策略订阅实现商业化
- **业务目标**: 
  - 展示丰富的交易策略
  - 帮助用户选择合适的策略
  - 建立策略订阅付费模式
- **前置条件**: 
  - 用户已登录
  - 策略广场已上线策略
- **后置条件**: 
  - 成功：用户订阅策略，可进行回测或实盘
  - 失败：订阅失败，提示原因
- **主要业务流程**:
  1. 用户进入策略广场
  2. 用户浏览策略列表（分类：趋势跟踪/网格交易/套利/AI策略等）
  3. 用户查看策略详情（描述、历史绩效、参数说明）
  4. 用户查看策略回测报告
  5. 用户选择订阅策略
  6. 系统展示订阅价格和条款
  7. 用户确认订阅（免费/付费）
  8. 系统添加策略到用户订阅列表
- **扩展流程**:
  - **E1.策略对比**: 用户可选择多个策略进行对比
  - **E2.策略收藏**: 用户可收藏感兴趣的策略
  - **E3.策略评分**: 用户可对使用过的策略评分
  - **E4.策略推荐**: 系统根据用户偏好推荐策略
- **业务规则**:
  - BR1: P0阶段仅展示系统自有策略
  - BR2: 策略绩效数据必须基于真实回测
  - BR3: 付费策略需明确收费标准
  - BR4: 订阅后可随时取消（按订阅周期）
- **量化指标**:
  - 策略详情页加载时间：<2秒
  - 策略订阅转化率：≥10%
  - 用户策略浏览深度：≥3个策略/session
- **优先级**: Must Have
- **来源**: （引用自用户需求描述"策略广场-包括系统自有的策略，P0只支持系统自有策略，策略用户可以订阅使用"）

### 业务用例4：策略回测验证

- **主要参与者**: 投资者（已订阅策略的用户）
- **利益相关者**: 
  - 投资者：希望在实盘前验证策略有效性
  - 平台运营：希望用户通过回测建立信心后进行实盘
  - 量化研究员：希望回测结果准确反映策略表现
- **业务目标**: 
  - 提供可靠的历史数据回测
  - 生成完整的策略绩效报告
  - 帮助用户评估策略风险收益
- **前置条件**: 
  - 用户已订阅策略
  - 系统有足够的历史数据
- **后置条件**: 
  - 成功：生成回测报告，包含绩效指标和交易明细
  - 失败：回测失败，提示原因
- **主要业务流程**:
  1. 用户选择已订阅的策略
  2. 用户配置回测参数：
     - 交易对（如BTC/USDT）
     - 回测时间范围
     - 初始资金
     - 策略参数（如有）
  3. 用户启动回测
  4. 系统按历史数据模拟策略运行
  5. 系统计算绩效指标
  6. 系统生成回测报告：
     - 收益曲线
     - 总收益率/年化收益率
     - 最大回撤
     - 夏普比率
     - 胜率
     - 盈亏比
     - 交易次数
     - 交易明细列表
  7. 用户查看和分析报告
- **扩展流程**:
  - **E1.参数优化**: 系统支持参数网格搜索，找出最优参数
  - **E2.多策略对比**: 用户可同时回测多个策略进行对比
  - **E3.不同周期对比**: 用户可对比不同时间段的回测结果
  - **E4.报告导出**: 用户可导出PDF/Excel格式的回测报告
- **业务规则**:
  - BR1: 回测必须使用真实历史数据
  - BR2: 回测考虑交易手续费（按交易所费率）
  - BR3: 回测考虑滑点模型
  - BR4: 禁止使用未来数据（避免前视偏差）
  - BR5: 回测时间范围：最长支持1年
- **量化指标**:
  - 回测执行时间：<30秒（1年数据）
  - 回测结果准确性：与实盘偏差<5%
  - 绩效指标覆盖率：100%
- **优先级**: Must Have
- **来源**: （引用自用户需求描述"用户可以用策略跑回测"）

### 业务用例5：策略实盘运行

- **主要参与者**: 投资者（已订阅策略且已配置交易所的用户）
- **利益相关者**: 
  - 投资者：希望策略自动执行交易
  - 风控团队：希望实盘交易符合风控规则
  - 平台运营：希望用户实盘活跃度高
- **业务目标**: 
  - 实现策略自动化交易
  - 保障实盘交易安全可靠
  - 提供实时监控和干预能力
- **前置条件**: 
  - 用户已订阅策略
  - 用户已配置交易所API Key（需要交易权限）
  - 用户账户有足够资金
  - 用户已了解风险并确认
- **后置条件**: 
  - 成功：策略开始运行，自动执行交易
  - 失败：启动失败，提示原因
- **主要业务流程**:

```python
from strategy_engine import Strategy, Signal

class MyMACrossStrategy(Strategy):
    """双均线交叉策略"""
    
    def __init__(self):
        self.fast_period = 10  # 快线周期
        self.slow_period = 20  # 慢线周期
    
    def on_bar(self, bar):
        # 计算均线
        fast_ma = self.sma(bar.close, self.fast_period)
        slow_ma = self.sma(bar.close, self.slow_period)
        
        # 生成信号
        if fast_ma > slow_ma and self.position <= 0:
            return Signal.BUY
        elif fast_ma < slow_ma and self.position > 0:
            return Signal.SELL
        
        return Signal.HOLD
```

##### 11.2.3 撮合逻辑规则
```
┌─────────────────────────────────────────────────────────────────┐
│                     模拟订单撮合逻辑                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  市价买单:                                                       │
│  ├─ 成交价格 = 当前卖一价（Ask Price）                           │
│  ├─ 滑点模拟（可选）= 成交价 * (1 + 滑点比例)                    │
│  └─ 立即成交，更新虚拟持仓                                       │
│                                                                 │
│  市价卖单:                                                       │
│  ├─ 成交价格 = 当前买一价（Bid Price）                           │
│  ├─ 滑点模拟（可选）= 成交价 * (1 - 滑点比例)                    │
│  └─ 立即成交，更新虚拟持仓                                       │
│                                                                 │
│  限价买单:                                                       │
│  ├─ 挂单等待                                                    │
│  ├─ 当 卖一价 ≤ 限价 时触发成交                                  │
│  └─ 成交价格 = 限价                                             │
│                                                                 │
│  限价卖单:                                                       │
│  ├─ 挂单等待                                                    │
│  ├─ 当 买一价 ≥ 限价 时触发成交                                  │
│  └─ 成交价格 = 限价                                             │
│                                                                 │
│  止损单（卖出）:                                                  │
│  ├─ 当 最新价 ≤ 止损价 时触发                                    │
│  └─ 触发后按市价单执行                                           │
│                                                                 │
│  止盈单（卖出）:                                                  │
│  ├─ 当 最新价 ≥ 止盈价 时触发                                    │
│  └─ 触发后按市价单执行                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 存储分层设计

```
┌─────────────────────────────────────────────────────────────────┐
│                      数据存储分层架构                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐        │
│  │   热数据     │    │   温数据     │    │   冷数据     │        │
│  │  (实时访问)  │    │ (近期访问)   │    │ (历史归档)   │        │
│  └──────┬──────┘    └──────┬──────┘    └──────┬──────┘        │
│         │                  │                  │                │
│         ▼                  ▼                  ▼                │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐        │
│  │   Redis     │    │ TimescaleDB │    │ Parquet/HDF5│        │
│  │  内存缓存    │    │  时序数据库   │    │  文件数据库   │        │
│  └─────────────┘    └─────────────┘    └─────────────┘        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 存储组件详细说明

| 存储类型 | 推荐方案 | 数据内容 | 说明 |
|---------|---------|---------|------|
| **关系数据库** | PostgreSQL | 用户信息、订单记录、策略配置、交易所配置 | ACID事务保证，成熟稳定 |
| **时序数据库** | **TimescaleDB** / **InfluxDB** | 实时行情、K线数据、Tick数据、指标数据 | 专为时序数据优化，高效压缩和查询 |
| **内存缓存** | **Redis** | 实时行情快照、会话状态、热点数据、订单簿 | 毫秒级读写，支持Pub/Sub行情推送 |
| **文件数据库** | **Parquet** / **HDF5** | 历史K线归档、回测数据集、大规模Tick数据 | 列式存储，高压缩比，批量读取高效 |
| **文档数据库** | MongoDB（可选） | 策略运行日志、非结构化配置 | 灵活Schema，适合日志类数据 |
| **搜索引擎** | Elasticsearch（可选） | 资讯全文搜索、日志检索 | 全文搜索，日志分析 |

#### 行情数据存储方案

| 数据类型 | 存储位置 | 保留策略 | 访问场景 |
|---------|---------|---------|---------|
| **实时Tick** | Redis (内存) | 最近1小时 | 实时行情展示、策略实盘 |
| **分钟K线** | TimescaleDB | 最近3个月（热）→ Parquet（冷） | 短周期分析、策略回测 |
| **小时K线** | TimescaleDB | 最近1年（热）→ Parquet（冷） | 中周期分析 |
| **日K线** | TimescaleDB + Parquet | 全量保留 | 长周期分析、历史回测 |
| **订单簿快照** | Redis | 最近深度数据 | 深度图展示、滑点分析 |
| **历史Tick** | Parquet/HDF5 | 全量归档 | 高精度回测 |

#### 时序数据库对比

| 数据库 | 类型 | 优势 | 劣势 | 推荐场景 |
|-------|------|------|------|---------|
| **TimescaleDB** | PostgreSQL扩展 | SQL兼容、成熟生态、自动分区 | 性能略低于专用时序库 | **首选**，与业务库统一 |
| **InfluxDB** | 专用时序库 | 写入性能极高、InfluxQL | 学习成本、单机版限制 | 超高频Tick数据 |
| **QuestDB** | 专用时序库 | 极致性能、SQL兼容 | 社区较小 | 高性能需求场景 |
| **ClickHouse** | OLAP列式库 | 大数据分析极快 | 运维复杂 | 海量历史数据分析 |

#### 文件数据库对比

| 格式 | 类型 | 优势 | 劣势 | 推荐场景 |
|-----|------|------|------|---------|
| **Parquet** | 列式存储 | 压缩率高、Spark/Pandas原生支持、Schema灵活 | 不支持追加写入 | **首选**，历史K线归档 |
| **HDF5** | 层次化存储 | Python生态好、支持追加、金融领域广泛 | 并发写入限制 | Tick数据、回测数据集 |
| **Feather/Arrow** | 内存映射 | 读取极快、跨语言 | 压缩率一般 | 策略回测快速加载 |
| **CSV** | 文本格式 | 简单通用 | 体积大、解析慢 | 数据导出、小规模备份 |

```
┌─────────────────────────────────────────────────────────────────────────────┐

│                                                                             │
│   交易所 ──WebSocket──▶ 行情网关 ──Kafka──▶ ┬─▶ Redis (实时缓存)           │
│   (Binance/OKX等)                          │                               │
│                                            ├─▶ TimescaleDB (时序存储)       │
│                                            │                               │
│                                            └─▶ 策略引擎 (实盘信号)          │
│                                                                             │
│   定时任务 ───────────────────────────────────────────────────────────────  │
│   │                                                                         │
│   ├─▶ TimescaleDB ──数据老化──▶ Parquet (历史归档，S3/OSS)                 │
│   │                                                                         │
│   └─▶ 聚合计算 ──────────────▶ 多周期K线生成                               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
