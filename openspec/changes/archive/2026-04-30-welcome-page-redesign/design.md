## Context

当前 Welcome 页面（`src/views/WelcomePage.vue`）是 Phase 1 实现的一部分，包含：
- 一个"快速连接"表单（host/port/password）+ "测试连接"按钮
- 测试成功后显示"保存为新连接"按钮
- 已保存连接列表（从 SQLite 加载）

核心问题：**没有"直接连接"能力**。用户必须先保存连接才能进入工作区，快速连接只能测试不能真正连接。这违背了产品设计文档中"5 秒内完成首次连接"的 NFR 要求。

后端已有完整的连接管理能力（`connection_open` 接收 connId 从 SQLite 加载配置），但缺少接收临时配置直接连接的命令。

## Goals / Non-Goals

**Goals:**
- 用户首次打开应用，填写 host/port/password 后一键连接进入工作区（≤ 5 秒）
- 已保存连接一键点击即可连接
- 新建连接流程完整：填写完整配置 → 测试 → 保存 → 自动连接
- 页面布局清晰区分"日常使用"（已保存连接）和"首次/临时使用"（快速连接）
- 临时连接进入工作区后可随时保存

**Non-Goals:**
- 不做连接导入/导出功能（P1 范围）
- 不做 SSH 隧道 / TLS 配置（P1 范围）
- 不做连接分组的拖拽排序
- 不做多连接同时打开（当前架构单连接）

## Decisions

### 1. 页面布局：响应式双区域

**选择**：采用左右分栏布局（已保存连接列表 | 快速连接 + 操作区），当无已保存连接时退化为居中单栏。

**替代方案**：
- Tab 切换（"已保存" / "新建"）—— 增加操作步骤，不如一目了然
- 纯列表 + 底部表单 —— 信息密度低，浪费空间

**理由**：左右分栏让两种场景并存，老用户左侧一键连接，新用户右侧快速上手。

### 2. 快速连接 = 真正连接（非仅测试）

**选择**：快速连接按钮改为"连接"，点击后直接建立 Redis 会话进入工作区。同时保留"测试连接"作为小字链接/次要按钮。

**替代方案**：
- 保持"测试连接" + 额外"连接"按钮 —— 两个按钮容易混淆
- 去掉测试功能 —— 用户有时只想验证连通性不想进入

**理由**：主操作应该是"连接"，测试是辅助。

### 3. 临时连接的后端实现

**选择**：新增 `connection_open_temp` IPC 命令，接收完整 `ConnectionConfig`（不含 id），在 ConnectionManager 中创建临时会话（使用 `temp-{uuid}` 作为 connId），不写入 SQLite。

**替代方案**：
- 复用 `connection_open`，先自动保存再连接 —— 污染用户的连接列表
- 前端先调 `connection_save` 再调 `connection_open` —— 两步操作，且用户可能不想保存

**理由**：临时连接是独立概念，不应强制持久化。用户可在连接后选择保存。

### 4. 连接成功后的保存提示

**选择**：临时连接进入工作区后，在 Statusbar 或 Toolbar 显示一个持久性的"保存连接"按钮/提示条，点击后打开 ConnectionForm 预填当前配置。

**替代方案**：
- Toast 提示 —— 会自动消失，用户可能错过
- 弹窗强制询问 —— 打断工作流

**理由**：非侵入式但持续可见，用户随时可以保存。

### 5. 已保存连接列表的交互增强

**选择**：
- 每个连接项显示：名称、host:port、分组标签、状态指示点
- 悬浮显示操作按钮（编辑/删除）
- 支持右键上下文菜单
- 按分组折叠展示（有分组时）
- 空状态显示引导插画 + "新建连接"按钮

## Risks / Trade-offs

- **[临时连接生命周期]** → 临时连接在应用关闭后丢失，用户如果忘记保存会丢失配置。通过持续可见的"保存"提示缓解。
- **[后端新增命令]** → 需要新增 `connection_open_temp` 命令，增加了 IPC 表面积。但逻辑简单（复用现有 `ConnectionManager::open`），风险可控。
- **[布局复杂度]** → 双区域布局在窗口较窄时可能拥挤。通过响应式设计（窄窗口时改为上下布局）缓解。
- **[向后兼容]** → 完全替换现有 WelcomePage.vue，但不影响其他组件和 store 的公开 API。
