## Context

Key 面板目前同时维护两类状态：当前详情 key（`selectedKey` / `currentKey`）和批量选择集合（`selectedKeys`）。普通点击会清空批量选择并打开详情，但不会建立 Shift 范围选择锚点；`rangeSelect()` 在没有锚点时退化为只切换当前点击 key，导致首次 Shift-click 选区错误。同时，批量选择时当前详情 key 仍显示 active 样式，视觉上像批量选中项，但它并不一定属于 `selectedKeys`。

## Goals / Non-Goals

**Goals:**

- 让普通点击后的首次 Shift-click 以普通点击的 key 为锚点，范围选择结果稳定可预期。
- 保持当前详情 key 与批量选择集合的语义分离：普通点击不进入批量模式。
- 批量模式下只突出显示真正属于 `selectedKeys` 的 key，避免 active 样式造成“也被选中”的误解。
- 在刷新、筛选、清空批量选择后维护有效的选择锚点，避免过期锚点影响下一次范围选择。

**Non-Goals:**

- 不重做 Key 面板布局或批量操作栏视觉设计。
- 不改变 Redis 批量删除、TTL 修改、复制名称的后端接口。
- 不改变右侧详情面板的数据加载与编辑能力。

## Decisions

1. 将范围选择锚点从“最后一个批量选中 key”语义调整为“最后一个可作为范围起点的 key”。
   - 普通点击设置当前详情 key，同时记录选择锚点。
   - Cmd/Ctrl-click 和 Shift-click 继续更新锚点。
   - `rangeSelect()` 优先使用锚点；锚点不存在或不在当前过滤列表中时，仅选择目标 key 并重置锚点，避免不可见 key 参与范围计算。
   - 备选方案：让普通点击直接把 key 放入 `selectedKeys`。放弃原因：这会让普通浏览详情也触发批量操作栏，违背“普通点击不进入批量模式”的目标。

2. 批量选择集合继续作为批量操作的唯一数据源。
   - 批量操作栏、数量、删除/TTL/复制均只读取 `selectedKeys`。
   - `selectedKey` 只表示右侧详情面板正在查看的 key，不作为批量操作依据。

3. 批量模式下抑制当前详情 key 的 active 选中样式。
   - 当 `selectedKeys.size > 0` 时，列表项 active 高亮不应与批量选中样式同时制造“选中感”。
   - 若当前详情 key 同时也在 `selectedKeys` 中，则只显示批量选中样式。
   - 这样不需要清空右侧详情状态，避免用户批量选择时右侧内容突然消失。

4. 选择锚点需要随列表上下文维护。
   - 清空批量选择时可保留或重新设置当前详情 key 作为锚点，确保下一次 Shift-click 可用。
   - 刷新、筛选后，如果锚点不在当前过滤列表，应清除锚点。

## Risks / Trade-offs

- [Risk] 用户可能期望普通点击也计入“已选 1 个键” → Mitigation：规格明确普通点击是浏览详情行为，只有修饰键选择才进入批量模式。
- [Risk] 隐藏 active 高亮后，批量模式下右侧详情与左侧列表的关联变弱 → Mitigation：保留右侧详情内容，只在退出批量选择后恢复 active 高亮。
- [Risk] 锚点与筛选列表不同步会造成范围错误 → Mitigation：范围选择前校验锚点是否存在于 `filteredKeys`，不存在则退化为目标 key 选择并重设锚点。
