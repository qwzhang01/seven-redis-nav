## 验收标准编写规范

### Given-When-Then 格式说明

```
Feature: [功能名称]
  Scenario: [场景名称]
    Given [前置条件]
    When [触发动作]
    Then [预期结果]
```

### 场景类型
- **Happy Path**: 正常业务流程
- **Edge Case**: 边界条件
- **Error Case**: 异常处理
- **Security Case**: 安全相关

### US064: 邮箱手机号注册

**Story Points**: 5 | **Priority**: Must Have

```gherkin
Feature: 用户注册
  作为新用户，我希望通过邮箱或手机号注册账户，以便开始使用量化交易系统

  Background:
    Given 用户在注册页面

  # Happy Path
  Scenario: 邮箱注册成功
    Given 用户选择邮箱注册方式
    When 用户输入有效邮箱 "user@example.com"
    And 用户输入密码 "Password123!"
    And 用户确认密码 "Password123!"
    And 用户点击获取验证码
    And 用户输入正确的邮箱验证码
    And 用户点击注册按钮
    Then 系统创建新账户
    And 用户收到注册成功提示
    And 用户自动登录系统

  Scenario: 手机号注册成功
    Given 用户选择手机号注册方式
    When 用户输入有效手机号 "+86 13800138000"
    And 用户输入密码 "Password123!"
    And 用户点击获取短信验证码
    And 用户输入正确的短信验证码
    And 用户点击注册按钮
    Then 系统创建新账户
    And 用户收到注册成功提示

  # Edge Cases
  Scenario Outline: 密码复杂度验证
    Given 用户输入邮箱 "user@example.com"
    When 用户输入密码 "<password>"
    Then 系统显示 "<message>"
    And 注册按钮 "<button_state>"

    Examples:
      | password      | message                    | button_state |
      | 123456        | 密码长度至少8位            | 禁用         |
      | password      | 密码需包含大写字母和数字   | 禁用         |
      | PASSWORD123   | 密码需包含小写字母         | 禁用         |
      | Password123!  | 密码符合要求               | 可用         |

  Scenario: 验证码60秒冷却
    Given 用户输入有效邮箱
    When 用户点击获取验证码
    Then 验证码按钮显示60秒倒计时
    And 倒计时期间按钮不可点击
    And 60秒后按钮恢复可点击

  # Error Cases
  Scenario: 邮箱已被注册
    Given 邮箱 "existing@example.com" 已被注册
    When 用户使用该邮箱注册
    Then 系统显示 "该邮箱已被注册"
    And 提供 "直接登录" 和 "找回密码" 链接

  Scenario: 验证码错误
    Given 用户已获取验证码
    When 用户输入错误验证码
    And 连续错误 <3> 次
    Then 系统显示 "验证码错误，请重新输入"
    And 用户可继续尝试

  Scenario: 验证码连续错误锁定
    Given 用户已获取验证码
    When 用户连续输入错误验证码 5 次
    Then 系统锁定该账户15分钟
    And 显示 "验证码错误次数过多，请15分钟后重试"
```

### 边界条件

| 边界场景 | 输入条件 | 预期行为 |
|---------|---------|---------|
| 邮箱格式边界 | `a@b.c` (最短有效格式) | 接受并继续 |
| 邮箱长度上限 | 254字符邮箱 | 接受；超过254字符拒绝 |
| 密码最小长度 | 7字符密码 | 拒绝，提示至少8位 |
| 密码最大长度 | 128字符密码 | 接受；超过128字符截断或拒绝 |
| 手机号格式 | +86 13800138000 | 接受中国大陆号码 |
| 国际手机号 | +1 2025551234 | 接受国际格式 |
| 验证码有效期 | 输入时距发送已过5分钟 | 拒绝，提示验证码已过期 |
| 验证码边界时间 | 输入时距发送4分59秒 | 接受 |
| 并发注册 | 两个请求同时提交相同邮箱 | 仅一个成功，另一个提示已注册 |

### US066: 重置密码

**Story Points**: 2 | **Priority**: Must Have

```gherkin
Feature: 重置密码
  作为忘记密码的用户，我希望重置密码，以便恢复账户访问

  Background:
    Given 用户在 "忘记密码" 页面

  # Happy Path
  Scenario: 邮箱重置密码成功
    When 用户输入注册邮箱 "user@example.com"
    And 用户点击发送重置链接
    Then 系统发送密码重置邮件
    And 显示 "重置链接已发送到您的邮箱"

  Scenario: 通过重置链接修改密码
    Given 用户收到重置邮件
    When 用户点击邮件中的重置链接
    And 用户输入新密码 "NewPassword456!"
    And 用户确认新密码
    And 用户点击确认
    Then 密码修改成功
    And 用户需重新登录

  # Edge Cases
  Scenario: 重置链接24小时过期
    Given 用户收到重置邮件已超过24小时
    When 用户点击重置链接
    Then 系统显示 "链接已过期，请重新获取"

  Scenario: 重置链接只能使用一次
    Given 用户已通过重置链接修改密码
    When 用户再次点击同一重置链接
    Then 系统显示 "链接已失效"

  # Error Cases
  Scenario: 未注册邮箱
    When 用户输入未注册的邮箱
    And 点击发送重置链接
    Then 系统显示 "已发送重置链接"
    But 实际不发送邮件（防止邮箱探测）
```

---

### 边界条件

| 边界场景 | 输入条件 | 预期行为 |
|---------|---------|---------|
| 链接有效期边界 | 23小时59分 | 链接有效 |
| 链接过期边界 | 24小时01分 | 链接过期 |
| 新密码与旧密码相同 | 输入旧密码 | 拒绝，提示不能与旧密码相同 |
| 重复发送限制 | 1分钟内再次请求 | 拒绝，提示稍后重试 |
| Token复用 | 同一Token使用两次 | 第二次拒绝 |

| 密码修改响应 | < 2秒 | API测试 |
| Token验证 | < 500ms | 后端性能监控 |

### US058: 添加交易所API Key

**Story Points**: 3 | **Priority**: Must Have

```gherkin
Feature: 添加交易所API Key
  作为交易用户，我希望添加交易所API Key，以便系统可以执行交易

    When 用户选择交易所 "Binance"
    And 用户输入API Key "abc123..."
    And 用户输入API Secret "xyz789..."
    And 用户输入备注名称 "我的主账户"
    And 用户点击保存
    Then 系统验证API Key有效性
    And 显示 "API Key 添加成功"
    And 新账户出现在账户列表

  Scenario Outline: 支持多交易所
    When 用户选择交易所 "<exchange>"
    Then 显示对应交易所的配置表单

    Examples:
      | exchange |
      | Binance  |
      | OKX      |
      | Bybit    |
      | Gate.io  |

  # Edge Cases
  Scenario: API Key权限检查
    Given 用户添加的API Key无交易权限
    When 用户保存API Key
    Then 系统提示 "该API Key无交易权限，仅支持只读操作"
    And 用户可选择继续添加或取消

  # Error Cases
  Scenario: API Key格式错误
    When 用户输入格式错误的API Key
    And 点击保存
    Then 系统显示 "API Key格式不正确"

  Scenario: API Key无效

    Then 系统显示 "API Key验证失败，请检查配置"

  # Security Cases
  Scenario: API Secret加密存储
    When 用户添加API Key成功
    Then API Secret使用AES-256加密存储
    And 列表中API Secret显示为 "****"
```

- [ ] API Secret 使用 AES-256 加密存储
- [ ] 密钥不明文记录在日志中

**Story Points**: 1 | **Priority**: Must Have

```gherkin
Feature: 删除交易所账户
  作为用户，我希望删除不再使用的交易所账户，以便保持账户列表整洁
```

```gherkin
Feature: 一键平仓
  作为投资者，我希望一键平掉所有持仓，以便紧急清仓止损

  Background:
    Given 用户有持仓的策略

  # Happy Path
  Scenario: 一键市价平仓
    When 用户点击 "一键平仓"
    Then 显示确认弹窗:
      | 内容                                   |
      | 将平掉以下持仓：                       |
      | BTC/USDT 多 0.1个 预估亏损-50 USDT     |
      | ETH/USDT 空 2个 预估盈利+80 USDT       |
      | 总持仓价值: 5000 USDT                  |
    When 用户输入交易密码确认
    Then 所有持仓以市价平仓
    And 相关策略自动暂停

  Scenario: 平仓执行结果
    Given 用户确认平仓
    Then 显示平仓进度
    When 平仓完成
    Then 显示结果:
      | 信息       | 值             |
      | 平仓数量   | 3笔            |
      | 实际成交   | 4950 USDT      |
      | 盈亏       | -30 USDT       |

  # Edge Cases
  Scenario: 部分平仓失败
    Given 某持仓平仓失败（交易所异常）
    Then 显示失败详情
    And 提供手动处理入口
    And 持续重试直到成功或超时
```

### US062: API异常自动暂停策略

**Story Points**: 5 | **Priority**: Must Have

```gherkin
Feature: API异常自动暂停
  作为系统，我希望API异常时自动暂停策略，以便保护用户资金安全

  Background:
    Given 策略正在运行

  # Happy Path
  Scenario Outline: API异常触发暂停
    Given 发生 "<exception>"
    When 异常持续超过 <threshold>
    Then 自动暂停相关策略
    And 发送告警通知用户

    Examples:
      | exception        | threshold |
      | API连接超时      | 30秒      |
      | API认证失败      | 立即      |
      | 下单连续失败5次  | 立即      |
      | 行情数据中断     | 60秒      |

  Scenario: 自动恢复机制
    Given 策略因API超时暂停
    When API恢复正常
    Then 系统自动尝试恢复策略
    And 恢复前验证账户状态
    And 恢复成功发送通知

  # Edge Cases
  Scenario: 手动恢复选项
    Given 用户配置为异常时不自动恢复
    When API异常后恢复正常
    Then 策略保持暂停状态
    And 等待用户手动恢复
```
